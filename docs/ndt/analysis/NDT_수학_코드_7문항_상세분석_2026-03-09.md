## Q1. d1, d2, d3는 어떻게 Approximation 하는가?

### 수학적 배경 (Magnusson 2009, Eq. 6.9–6.10)

NDT에서 한 포인트의 정합 score를 **혼합 확률 모델(Mixture Model)** 로 정의한다:

```
p(x) = c1 * exp(-d2/2 * x^T Σ^{-1} x)  +  c2
       ──────────────────────────────────────────
       인라이어 Gaussian                   아웃라이어 균등분포
```

- `c1 = 10 * (1 - outlier_ratio)`: 인라이어 Gaussian의 진폭 (스케일)
- `c2 = outlier_ratio / resolution³`: 아웃라이어의 균등 밀도 (복셀 부피당 확률)
- `x^T Σ^{-1} x`: Mahalanobis 거리 (정렬 오차의 가중 제곱합)

이 혼합 모델의 **log-likelihood를 최대화**하면, 정규화 후:

```
s(x) = -d1 * exp(-d2/2 * x^T Σ^{-1} x)
```

### d3, d1, d2 유도

**d3** (내부 정규화 오프셋):
```
d3 = -log(c2)
```
아웃라이어 항을 log-scale에서 0으로 이동시키는 역할. 최종 score 식에는 등장하지 않는다.

**d1** (score 진폭 — 음수):
```
d1 = -log(c1 + c2) - d3
   = -log(c1 + c2) + log(c2)
   = log(c2 / (c1 + c2))
```
`c1 > 0, c2 > 0` 이면 `c2 / (c1+c2) < 1` → **d1 < 0**

예: `outlier_ratio = 0.1, resolution = 1.0`일 때:
- `c1 = 9.0`, `c2 = 0.1`
- `d1 = log(0.1 / 9.1) ≈ -4.51`  
- `-d1 ≈ 4.51 > 0` → score `= -d1 * e_term > 0` ✓

**d2** (Mahalanobis 기울기 — 양수):
```
d2 = -2 * log( (-log(c1 * exp(-0.5) + c2) - d3) / d1 )
```
이것은 Mahalanobis 거리 `m = x^T Σ^{-1} x = 1` (즉, 1σ 떨어진 거리)일 때
score가 얼마나 되어야 하는지를 **역산**하여 구한다:

```
s(m=1) = -d1 * exp(-d2/2)
       = c1 * exp(-0.5) / (c1 + c2) 의 정규화 버전
```

→ 이를 d2에 대해 풀면 위의 공식이 나온다.

### 코드 (integrated_ndt_factor.hpp:67-73)

```cpp
inline void compute_ndt_params(double resolution, double outlier_ratio, double& d1, double& d2) {
  double c1 = 10.0 * (1.0 - outlier_ratio);
  double c2 = outlier_ratio / (resolution * resolution * resolution);
  double d3 = -std::log(c2);          // 내부용 정규화 항
  d1 = -std::log(c1 + c2) - d3;      // 음수
  d2 = -2.0 * std::log((-std::log(c1 * std::exp(-0.5) + c2) - d3) / d1);  // 양수
}
```

### 호출 시점

`update_correspondences()`에서 **매번** 호출된다 (hpp:119):

```cpp
compute_ndt_params(resolution, outlier_ratio, gauss_d1, gauss_d2);
```

사실 `resolution`과 `outlier_ratio`가 바뀌지 않으면 같은 값이 나오므로, 캐싱으로 최적화 가능하다. 현재는 매 대응점 탐색마다 재계산한다.

## Q3. Jacobian이 왜 `<4, 6>`으로 계산되는가?

### 이유 1: 포인트가 4D Homogeneous 좌표

NDT에서 포인트는 `Eigen::Vector4d`로 저장된다:

```cpp
const auto& mean_A = frame::point(*source, i);  // Vector4d: (x, y, z, 1)
Eigen::Vector4d transed_mean_A = delta * mean_A; // Vector4d
Eigen::Vector4d residual = mean_B - transed_mean_A; // Vector4d
```

`residual`이 4차원이므로 `∂residual/∂pose`는 자연히 `4 × DOF`가 된다.

### 이유 2: SE(3) Lie Algebra가 6DOF

`delta ∈ SE(3)`. Lie algebra `ξ ∈ ℝ⁶`:

```
ξ = [ω₁, ω₂, ω₃, t₁, t₂, t₃]^T
    ───────────────  ──────────
    rotation (3)    translation (3)
```

따라서 pose `ξ`에 대한 점 변환의 Jacobian은:

```
∂(T·p) / ∂ξ  →  ℝ⁴ × ℝ⁶ = 4×6 행렬
```

### 실제 구조 (J_target 기준)

```
J = [ -Hat(q)   I₃  ] ← 3×3ブロック + 3×3블록
    [   0  0  0  0 0 0 ]  ← 마지막 행 (동차좌표, 항상 0)
```

- 행 0~2 (실제 xyz): 유효한 Jacobian 값
- 행 3 (동차 w=1): 항상 0 (변환에 의해 w는 변하지 않음)

코드에서 4×6 전체를 사용하지만 마지막 행은 0이므로 3×6이 실질적이다. 4×6을 쓰는 이유는 `inv_cov_B`가 4×4이고 `residual`이 Vector4d이기 때문에 차원을 맞추기 위해서다:

```cpp
// weight * J^T * inv_cov * J
// (6×4) * (4×4) * (4×6) = (6×6)  ← H에 누적
```

## Q4. `J_target`, `J_source`는 왜 각각 `transed_mean_A`, `mean_A`로 하고 이것들이 어떤 좌표의 점인가?

### 좌표계 정의

이 팩터는 **binary factor**: 두 개의 최적화 변수 `T_target`, `T_source`에 대해 동시에 미분한다.

```
delta = T_target^{-1} * T_source   (상대 변환)
transed_mean_A = delta * mean_A   (= T_target^{-1} * T_source * mean_A)
```

| 변수 | 좌표계 | 설명 |
|-----|--------|-----|
| `mean_A` | **source 프레임** (로컬 좌표) | 소스 포인트의 원래 좌표 |
| `transed_mean_A` | **target 프레임** (global 또는 target body 좌표) | delta로 변환된 소스 포인트 |
| `mean_B` | **target 프레임** | target 복셀의 평균 |

### J_target 유도 (`T_target`에 대한 Jacobian)

`T_target`이 perturbation `δξ_target` 만큼 변할 때:

```
delta' = (T_target * exp(δξ_target))^{-1} * T_source
       = exp(-δξ_target) * T_target^{-1} * T_source
       = exp(-δξ_target) * delta
```

따라서:
```
transed_mean_A' = exp(-δξ_target) * delta * mean_A
                ≈ (I - Hat(δω)) * transed_mean_A
```

1차 근사:
```
∂transed_mean_A / ∂δξ_target = [-Hat(transed_mean_A.xyz)  |  I₃]
```

→ **변환된 좌표** `transed_mean_A`에서 정의됨 (target 프레임)

```cpp
J_target.block<3,3>(0,0) = -gtsam::SO3::Hat(transed_mean_A.head<3>());  // rotation
J_target.block<3,3>(0,3) = Eigen::Matrix3d::Identity();                 // translation
```

### J_source 유도 (`T_source`에 대한 Jacobian)

`T_source`가 perturbation `δξ_source` 만큼 변할 때:

```
delta' = T_target^{-1} * T_source * exp(δξ_source)
```

따라서:
```
transed_mean_A' = T_target^{-1} * T_source * exp(δξ_source) * mean_A
                ≈ delta * (I + Hat(δω)) * mean_A
                = transed_mean_A + delta.linear() * Hat(δω) * mean_A
```

1차 근사:
```
∂transed_mean_A / ∂δξ_source = [delta.linear() * Hat(mean_A.xyz)  |  -delta.linear()]
```

→ **원래 소스 좌표** `mean_A`에서 정의됨 (source 프레임)

```cpp
J_source.block<3,3>(0,0) = delta.linear() * gtsam::SO3::Hat(mean_A.head<3>());  // rotation
J_source.block<3,3>(0,3) = -delta.linear();                                     // translation
```

### 직관적 이해

| | J_target | J_source |
|---|---------|---------|
| 미분 대상 | target pose `T_target` | source pose `T_source` |
| 기준 점 | `transed_mean_A` (target 프레임) | `mean_A` (source 프레임) |
| rotation 블록 | `-Hat(q)` | `R * Hat(a)` |
| translation 블록 | `I` | `-R` |
| 부호 차이 이유 | target을 움직이면 상대변환이 반대로 | source를 움직이면 상대변환이 그대로 |

## Q5. cost에서 `-d1`을 빼야 하는가? (수학적/코드적 논의)

### 원본 Magnusson score

Magnusson 2009에서 score는 **최대화** 목표:

```
s(x) = -d1 * exp(-d2/2 * m),   d1 < 0 이므로 s > 0
```

정렬이 잘 될수록 `m → 0`, `exp → 1`, `s → -d1 > 0` (최대값).

### GTSAM 비용 함수의 요구사항

GTSAM LM optimizer는 **최소화** 방향으로 동작하며, 비용(cost) 값이:
1. **비음수(≥ 0)** 이어야 한다.
2. 정렬이 좋을수록 **감소**해야 한다.
3. **완벽 정렬** 시 비용이 0이어야 한다.

### cost = `-d1 * e_term` 방식의 문제점

```
cost = -d1 * e_term = |d1| * exp(-d2/2 * m)
```

- `m = 0` (완벽 정렬): `cost = |d1| > 0` → 최솟값이 0이 아님!
- `m → ∞` (최악): `cost → 0`

→ **cost가 거꾸로다**: 정렬이 좋을수록 cost가 크고, 나쁠수록 0에 수렴한다. LM optimizer는 이것을 "이미 최적화된 상태"로 잘못 판단하게 된다.

### cost = `-d1 - (-d1 * e_term)` = `-d1 * (1 - e_term)` 방식 (현재 코드)

```cpp
const double score_function = -gauss_d1 * e_term;   // 양수
const double cost = -gauss_d1 - score_function;      // = -d1 * (1 - e_term)
```

| 상황 | e_term | cost |
|------|--------|------|
| 완벽 정렬 (m=0) | 1.0 | 0 ✓ |
| 부분 정렬 (m=1) | ~0.6 | `|d1|*0.4 > 0` ✓ |
| 최악 정렬 (m→∞) | 0 | `|d1| > 0` (최대) ✓ |

→ 비음수이고, 정렬이 좋을수록 감소하며, 완벽 정렬 시 0.

### `-d1` 상수항이 gradient에 미치는 영향

```
∂cost/∂pose = ∂(-d1 * (1 - e_term)) / ∂pose
            = ∂(-d1 - score_function) / ∂pose
            = 0  -  ∂(score_function) / ∂pose
```

`-d1`은 상수이므로 **gradient에 기여하지 않는다**. Jacobian/Hessian도 동일하다.

하지만 `-d1`이 없으면:

```
cost_wrong = d1 * e_term   (d1 < 0이므로 음수!)
```

GTSAM이 "cost = 음수" 상황을 만나면 정규화 실패 또는 이상 수렴한다.

### 결론

`-d1` 오프셋 항은:
- **gradient/Hessian에는 영향 없음** (최적화 방향 동일)
- **cost 값의 비음수성 보장**에 필수
- **LM의 λ 조정 로직**(cost 감소 여부 판단)에 필요

→ **빼면 안 된다**.

## Q6. weight 계산식이 맞고, 왜 루트를 씌우면 빨라지는가? 제곱을 하면 어떻게 되는가?

### 현재 weight 계산식

```cpp
const double weight = -gauss_d1 * gauss_d2 * e_term;
```

**수학적 유도**:

```
cost(m) = -d1 * (1 - exp(-d2/2 * m))   where m = r^T Σ^{-1} r

∂cost/∂m = -d1 * d2/2 * exp(-d2/2 * m)

∂²cost/∂m² = (-d1) * (d2/2)² * exp(-d2/2 * m)
```

GN 근사에서 Hessian은:

```
H ≈ J^T * (∂²cost/∂r²) * J
  = J^T * (weight_scalar * Σ^{-1}) * J
```

여기서 `weight_scalar = d(score)/d(e_term) * (d2/2)` 를 정리하면:

```
weight = -d1 * d2 * e_term
```

(코드에서는 `d2/2` 대신 `d2` 전체를 쓰고 `J^T * Σ^{-1} * J`에서 이미 Σ^{-1}가 있으므로 스케일 차이가 있음. 중요한 건 부호와 e_term 의존성.)

### weight는 항상 양수인가?

- `-d1 > 0` (d1이 음수이므로)
- `d2 > 0`
- `e_term = exp(...) > 0`

→ **weight = (-d1) * d2 * e_term > 0** 항상 양수 ✓

→ H = `weight * J^T * Σ^{-1} * J`는 **PSD(양반정치)** 보장 ✓

### 루트(√m)를 씌우면 왜 빨라지는가?

이것은 코드 최적화 문제다. `update_correspondences`에서 이웃 복셀 중 **최소 Mahalanobis 거리**를 찾을 때:

```cpp
double mahalanobis_dist = diff.transpose() * inv_cov * diff;
if (mahalanobis_dist < min_mahalanobis) { ... }
```

현재 코드는 `m` 자체를 비교한다. 만약 `√m`(Mahalanobis norm)을 사용하면:

```cpp
double mahal_norm = std::sqrt(mahalanobis_dist);
if (mahal_norm < min_mahal_norm) { ... }
```

`√m₁ < √m₂ ⟺ m₁ < m₂` 이므로 **결과는 동일**하지만 `sqrt` 연산이 추가된다.

역으로, `√m`를 이미 써왔다면 `m`으로 바꾸면 `sqrt` 연산을 **제거**하여 빨라진다.

**결론**: 현재 코드는 이미 `m` 직접 비교를 쓰고 있어서 최적화된 상태다. `√m`로 바꾸면 오히려 느려진다.

### 제곱(m²)을 쓰면 어떻게 되는가?

**대응점 탐색** 시 `m²`로 비교하면:

```
m₁² < m₂²  ⟺  |m₁| < |m₂|  (m > 0이므로 동일)
```

탐색 결과는 같다. 하지만 **evaluate()에서 `m²`를 Mahalanobis 거리로 사용하면**:

```
e_term = exp(-d2/2 * m²)   [잘못된 사용]
```

`m = 1` → `exp(-d2/2)` (동일)  
`m = 2` → `exp(-d2/2 * 4) = exp(-2*d2)` vs 올바른 `exp(-d2)` 

→ **m이 클수록 e_term이 훨씬 빠르게 0에 수렴**한다. 즉:
- 정렬이 조금만 나빠져도 weight ≈ 0 → gradient ≈ 0 → **달팽이 수렴**
- "Flat Landscape" 문제가 심화됨

→ `m²`는 수렴 속도를 **악화**시킨다.

### weight가 e_term에 의존하는 효과 (exponential decay)

```
weight = -d1 * d2 * exp(-d2/2 * m)
```

`m`이 크면(정렬 불량) weight가 지수적으로 0에 수렴. 이것이 NDT의 구조적 약점이기도 하다. 초기 정렬이 매우 나쁜 경우:
- 모든 포인트의 weight ≈ 0
- Hessian ≈ 0
- gradient ≈ 0
- LM step ≈ 0 → **수렴 불가**

## Q7. 왜 지금 코드가 GTSAM LM 구조와 잘 안 맞아서 수렴이 느린가?

### 배경: GTSAM LM의 작동 방식

GTSAM LM(Levenberg-Marquardt)은:

1. **비용 함수**: 각 factor의 `error()` 반환값의 제곱합으로 전체 비용 계산
2. **선형화**: 각 factor의 `linearize()` (= `evaluate()` 호출)로 H, b 얻음
3. **Hessian 양정치 가정**: Cholesky 분해로 `H · δx = -b` 풀기 → H가 PSD여야 함
4. **λ 조정**: step이 실제로 비용을 줄이면 λ 감소(GN 방향), 아니면 λ 증가(gradient descent 방향)

### 문제 1: H1-only GN 근사의 부정확성

NDT 비용 함수의 정확한 Hessian은:

```
H_exact = H1 + H2 + H3
```

- **H1** (GN 항): `weight * J^T * Σ^{-1} * J` ← 현재 코드가 쓰는 것
- **H2** (1차 도함수 항): `weight * d2 * (J^T * Σ^{-1} * r) * (r^T * Σ^{-1} * J)` 
- **H3** (잔차 항): `weight * J_r^T * Σ^{-1} * J_r` (Jacobian의 Jacobian)

H2, H3를 생략하므로:

```
H_approx << H_exact   (근사 Hessian이 실제보다 작다)
```

LM은 이 H_approx를 바탕으로 step size를 결정하는데, **실제 곡률보다 flat하다고 착각**하여 step을 과소평가 → **수렴 속도 저하**

**단, H1-only를 쓰는 이유**: H2 포함 시 indefinite(비양정치)가 될 수 있어 Cholesky 실패 가능성 있음.

### 문제 2: Exponential Weighting에 의한 Flat Landscape

```
weight = -d1 * d2 * exp(-d2/2 * m)
```

정렬이 나쁠수록 weight가 지수적으로 0에 가까워진다.

```
m = 4 → weight ≈ -d1 * d2 * exp(-2) ≈ 0.135 * (-d1 * d2)
m = 9 → weight ≈ -d1 * d2 * exp(-4.5) ≈ 0.011 * (-d1 * d2)
```

초기 정렬 불량 시: **거의 모든 포인트의 gradient가 0** → 사실상 첫 iteration에서 이미 수렴 불가.

### 문제 3: NDT Score의 비볼록성(Non-Convexity)

NDT score 함수는 비선형이며:
- 지역 최솟값이 많음
- 넓은 평탄 구간 존재 (flat basin)

GN 근사 Hessian(H1-only)은 이 비볼록 구조를 전혀 포착하지 못한다. LM의 λ는 H1-only에 기반하므로 실제 필요한 정규화보다 약함.

### 문제 4: GTSAM factor 구조와의 불일치

GTSAM의 일반적인 factor(`NoiseModelFactor` 등)는:

```
e(x) = f(x) - z       (잔차 벡터)
cost = 0.5 * e^T * Ω * e   (Mahalanobis norm의 제곱)
```

이 구조에서 H, b는 정확히:
```
H = J^T * Ω * J
b = J^T * Ω * e
```

NDT의 `evaluate()`는 이 구조를 우회하여 **직접 H, b를 계산하고 누적**한다. 즉:

- `IntegratedMatchingCostFactor`의 특수 경로 사용
- GTSAM 내부 정규화 로직과 반드시 일치하지 않음
- weight scaling이 GTSAM의 `noiseModel` 기반 스케일링과 다름

### 정리: 수렴이 느린 원인 요약

| 원인 | 구체적 메커니즘 | 영향 |
|------|--------------|------|
| H1-only GN | 실제 Hessian보다 작은 H → step 과소평가 | iter 수 증가 |
| Exponential weight decay | 초기 정렬 불량 시 gradient ≈ 0 | 달팽이 수렴 또는 발산 |
| 비볼록 비용 함수 | 지역 최솟값, flat landscape | LM λ 조정 실패 |
| factor 구조 불일치 | GTSAM 정규화와 weight scaling 불일치 | lambda 튜닝 효과 감소 |

### 비교: LightNDT가 빠른 이유

LightNDT(ndt_omp 계열)는:
1. **score 기반 대신 least-squares 형태**: `e = r`, `cost = 0.5 * r^T Σ^{-1} r`
2. **weight 없음** (또는 `weight = 1`): Hessian이 항상 양수적이고 step이 크다
3. **Gauss-Newton 직접 수행**: Flat landscape 문제 없음

→ NDT score 방식 대비 초기 수렴이 훨씬 빠르지만, 아웃라이어 강건성은 낮다.
